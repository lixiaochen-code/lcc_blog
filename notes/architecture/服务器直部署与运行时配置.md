# 服务器直部署与运行时配置

当前方案默认不依赖 Docker，而是直接在服务器上运行 Node 进程和现有知识库脚本。

## 部署原则

- 保持仓库结构简单，不引入 Docker 作为前置依赖。
- 生产站点仍然可以是静态站点。
- AI 相关能力通过服务器进程提供，而不是塞进静态页面本身。
- 知识库源码、运行时配置、发布动作保持分层。

## 推荐部署形态

建议拆成两个部分：

### 1. 静态站点

- 由 `pnpm docs:build` 产出 `.vitepress/dist`
- 可部署到 Nginx、Caddy 或任意静态文件服务

### 2. AI 服务进程

- 直接运行在服务器上
- 负责用户鉴权、权限校验、Token 配额控制
- 负责调用 MCP、skills、`kb:agent`
- 负责生成待发布请求

这意味着：

- 浏览站点本身不需要 Node 常驻进程
- 只有 AI 工具和管理后台需要服务器进程支持

## 为什么不强依赖 Docker

当前项目规模不大，且已经具备稳定的本地脚本体系：

- [`scripts/kb/agent.mjs`](/Users/apple/Documents/ai/lcc_blog/scripts/kb/agent.mjs)
- [`scripts/kb/url.mjs`](/Users/apple/Documents/ai/lcc_blog/scripts/kb/url.mjs)
- [`scripts/kb/build.mjs`](/Users/apple/Documents/ai/lcc_blog/scripts/kb/build.mjs)

对于这类场景，直接部署有几个现实优势：

- 上手快
- 维护成本低
- 配置更直观
- 更适合单人 owner 控制

后续如果用户量明显上升，再考虑引入容器化即可。

## 运行时配置文件

建议分成两个文件：

- [`data/ai-runtime.example.json`](/Users/apple/Documents/ai/lcc_blog/data/ai-runtime.example.json)：可提交到仓库的模板
- `data/ai-runtime.local.json`：服务器本地私有配置，不提交仓库

当前已经约定：

- `data/ai-runtime.local.json` 已加入 `.gitignore`
- 实际密钥仅保存在服务器本地文件中

## owner 可管理的运行时字段

最高权限用户可以修改：

- `protocol`
- `baseUrl`
- `apiKey`
- `model`

其中模型必须遵守一个约束：

- 系统同一时刻只有一个生效模型
- 非 owner 用户不能切换模型
- 普通用户只能使用 owner 已选定的模型

## 配置命令

当前仓库提供了一个轻量命令入口：

- `pnpm kb:runtime-config --action inspect`
- `pnpm kb:runtime-config --action set --protocol https --baseUrl api.openai.com/v1 --model gpt-4.1-mini`

如果要设置密钥，可以额外传入：

- `pnpm kb:runtime-config --action set --apiKey "<your-key>"`

该命令的职责是：

- 读取当前运行时配置
- 更新服务器协议和上游地址
- 更新唯一生效模型
- 更新本地保存的 API Key

## 运行时配置建议结构

建议配置至少包含：

- `server.protocol`
- `server.baseUrl`
- `model.selected`
- `credentials.apiKey`
- `permissions.ownerCanManageConnection`
- `permissions.ownerCanManageModel`

这样后续无论是后台页面还是 API，都可以围绕这一份配置工作。

## 安全建议

- `apiKey` 只保存到服务器本地文件或受保护的环境变量。
- 前端页面不要直接暴露完整 key。
- 配置查询接口默认返回脱敏后的 key。
- 修改 `baseUrl`、`protocol`、`apiKey`、`model` 必须只允许 owner 执行。

## 与权限系统的关系

运行时配置属于系统级能力，不属于普通知识库编辑权限。

因此建议把这类权限单独建模，例如：

- `runtime.manage_connection`
- `runtime.manage_model`
- `runtime.manage_secret`

默认只赋予 owner。

## 当前落地建议

接下来做应用层时，优先级建议如下：

1. 增加用户与权限表
2. 增加运行时配置读写接口
3. 增加 owner 管理页面
4. 增加 AI 动作网关
5. 最后再接入前端聊天窗口
